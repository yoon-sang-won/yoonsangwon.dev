# 콘텐츠 편집 원칙

## 블로그의 목적

이 사이트는 개발, UI, 웹 접근성, 반응형 구현, 작업 방식과 AI 활용에 관해 직접 배우고 시도한 내용을 오래 남기는 개인 블로그이자 학습 아카이브다. 이력서, 채용 포트폴리오, 경력 전시나 성과 홍보를 목적으로 하지 않는다.

글의 가치는 화려한 결론보다 당시의 질문, 확인한 근거, 직접 한 시도와 남은 한계를 정직하게 남기는 데 있다. 실제 경험하지 않은 사건, 성과, 수치와 타인의 정보를 채워 넣지 않는다.

## Writing, Notes, Lab 구분

### Writing

하나의 주제를 충분히 설명해 처음 보는 사람이 글 하나만 읽어도 맥락과 결론을 이해할 수 있을 때 사용한다.

- 문제와 배경, 판단 과정, 배운 점이 연결된다.
- 단순한 링크 모음이나 코드 조각보다 설명이 중심이다.
- 시간이 지난 뒤에도 다시 읽을 가치가 있는 정리 글이다.
- 완벽한 정답보다 현재 근거로 내린 판단과 한계를 분명히 쓴다.

### Notes

짧은 발견, 학습 메모, 명령어나 코드 조각, 문제 해결 단서와 아직 완결되지 않은 생각을 빠르게 보존할 때 사용한다.

- 한 가지 사실이나 해결 방법만 기록해도 된다.
- 긴 서론과 완결된 결론을 요구하지 않는다.
- 나중에 Writing이나 Lab의 재료가 될 수 있다.
- 출처와 실행 조건이 중요하면 짧더라도 함께 남긴다.

### Lab

직접 실행한 구현 실험, 비교, 실패와 시행착오를 재현 가능한 범위에서 기록할 때 사용한다.

- 확인하려는 질문과 실험 조건이 있다.
- 무엇을 시도했고 실제 결과가 어땠는지 구분한다.
- 성공뿐 아니라 실패, 예상과 달랐던 점과 한계를 남긴다.
- 필요한 경우 demo나 source 링크를 본문 문맥 안에 넣는다.

## 분류가 애매할 때

다음 순서로 판단한다.

1. 직접 실행한 조건과 결과를 비교하는 글인가? 그러면 Lab이다.
2. 독립적으로 읽을 수 있는 설명과 판단을 갖춘 글인가? 그러면 Writing이다.
3. 짧은 발견, 해결 단서 또는 미완성 생각을 우선 보존하려는가? 그러면 Notes다.
4. 아직 판단하기 어렵다면 Notes의 `draft: true`로 시작한다. 내용이 커진 뒤 파일을 옮기면 URL이 달라질 수 있으므로 공개 전에 최종 분류를 확정한다.

분량만으로 분류하지 않는다. 짧아도 실험 기록이면 Lab일 수 있고, 길어도 정리되지 않은 메모라면 Notes일 수 있다.

## 제목 작성 기준

- 본문이 실제로 답하는 질문이나 다루는 대상을 드러낸다.
- 결과를 과장하거나 클릭을 유도하는 표현을 쓰지 않는다.
- “완벽한”, “무조건”, “혁신적인”, “놀라운”처럼 근거가 필요한 수식어를 습관적으로 붙이지 않는다.
- 같은 주제의 글과 구분될 정도로 구체적으로 쓴다.
- 날짜나 버전이 핵심 조건이면 제목 또는 첫 문단에 밝힌다.
- 제목을 바꿔도 공개 URL인 파일명은 바꾸지 않는다.

## Description 작성 기준

Writing과 Lab에는 필수이고 Notes에는 필요할 때 작성한다.

- 한두 문장으로 무엇을 다루고 독자가 무엇을 확인할 수 있는지 설명한다.
- 제목을 그대로 반복하지 않는다.
- 본문에 없는 결론, 성과와 약속을 넣지 않는다.
- 검색어를 나열하지 않고 자연스러운 문장으로 쓴다.
- “이 글에서는 알아봅니다” 같은 빈 문장보다 구체적인 문제와 범위를 적는다.

## 태그 사용 기준

현재 콘텐츠 schema와 Pages CMS에는 태그 필드가 없다. 따라서 frontmatter에 `tags`를 임의로 추가하지 않고, 태그를 전제로 한 분류나 검색을 운영하지 않는다.

공개 글이 약 30개 이상 쌓이고 Writing·Notes·Lab만으로 반복해서 찾기 어려워졌을 때 도입을 검토한다. 실제 글에서 5~10개의 안정적인 주제어를 먼저 추출한 뒤 Astro schema, Pages CMS와 출력 화면을 한 번에 변경한다. 도입 전에는 본문 소제목, 명확한 제목과 관련 글 링크로 주제를 드러낸다. 자세한 기준은 [콘텐츠 모델](content-model.md#11-tags-도입-기준)을 따른다.

## 게시일과 수정일

- `published`는 글을 처음 공개한 날짜다. 초안을 처음 저장한 날짜가 아니다.
- `updated`는 결론, 코드, 실험 조건이나 설명처럼 독자의 이해에 영향을 주는 변경에만 입력한다.
- 오탈자, 띄어쓰기, 링크 교체와 가벼운 문장 다듬기만으로 수정일을 바꾸지 않는다.
- 수정일은 게시일과 같거나 이후여야 한다.
- 수정했다고 게시일을 바꾸거나 최신 글처럼 다시 올리지 않는다.
- 날짜는 `YYYY-MM-DD` 형식을 사용한다.

## 기존 글을 수정할 때

1. 기존 URL과 파일명을 유지한다.
2. 당시의 판단을 현재 지식으로 조용히 덮어쓰지 않는다. 중요한 변화는 무엇이 달라졌는지 밝힌다.
3. 인용과 외부 링크가 여전히 원문을 뒷받침하는지 확인한다.
4. 코드와 절차는 현재도 실행되는지, 특정 버전에만 해당하는지 확인한다.
5. 의미가 바뀌는 수정에는 `updated`를 기록한다.
6. 저장 후 Git diff에서 의도하지 않은 frontmatter나 Markdown 재포맷이 없는지 확인한다.

## 오래된 정보 고지

일부 내용만 오래됐고 기록 가치가 남아 있으면 삭제하지 않고 본문 도입부 뒤에 다음처럼 알린다.

```md
> 2027-03-15 안내: 이 글은 Astro 7 기준으로 작성했습니다. 현재 버전에서는 설정 방법이 달라질 수 있으며, 최신 공식 문서를 함께 확인하세요.
```

핵심 결론이 바뀌었다면 무엇이 달라졌는지와 최신 글 또는 공식 문서 링크를 함께 제공한다. 수정으로 현재 기준에 맞게 다시 검증했다면 `updated`도 변경한다. 단순히 오래됐다는 이유만으로 과거의 실험 결과를 현재 결과처럼 고쳐 쓰지 않는다.

## 삭제보다 draft 전환을 우선할 때

다음 경우에는 파일을 삭제하기보다 `draft: true`로 전환한다.

- 사실 오류나 민감 정보 여부를 다시 확인해야 한다.
- 링크가 깨졌거나 코드가 오래돼 임시로 공개를 멈춰야 한다.
- 구조를 크게 고치거나 다른 글과 통합할 예정이다.
- 이미 공개된 URL과 Git 기록을 보존할 필요가 있다.

중복 파일, 실수로 만든 테스트 글, 공개할 가치가 없고 참조도 없는 초안처럼 완전히 폐기할 대상만 삭제한다. 개인정보나 비밀정보가 Git 기록에 들어갔다면 draft 전환만으로 숨겨지지 않는다. 즉시 노출 범위를 확인하고 [복구 안내](recovery-guide.md)에 따라 별도 대응한다.

## 이미지 운영 규칙

### 폴더

첫 버전은 모든 콘텐츠 이미지에 다음 단일 폴더를 사용한다.

```text
public/images/content/
```

Pages CMS는 업로드 파일명을 무작위 안전 이름으로 바꾼다. 글별 자동 하위 폴더 기능은 구현하지 않으며, 이미 공개된 이미지를 폴더 정리만을 위해 이동하지 않는다. 이동하면 Markdown URL도 함께 바뀐다.

### 파일명과 형식

- 직접 파일명을 준비할 때는 영문 소문자, 숫자와 하이픈을 사용해 `slug-purpose-01.webp`처럼 작성한다.
- Pages CMS가 랜덤 이름으로 바꾼 파일은 저장 후 다시 이름을 바꾸지 않는다.
- 사진과 일반 화면 캡처는 WebP를 우선한다.
- 투명 배경이나 선명한 픽셀 표현이 꼭 필요하면 PNG를 사용한다.
- 움직임이 반드시 필요하지 않다면 GIF를 사용하지 않는다.
- SVG는 업로드하지 않는다.

### 크기와 alt

- 긴 변 기준 1600px 이하, 파일 하나당 1MB 이하를 기본 목표로 한다.
- 모바일 화면에서도 글자를 읽어야 하는 캡처는 필요한 부분만 잘라내고 실제 모바일 미리보기에서 확인한다.
- alt에는 이미지가 글에서 전달하는 핵심 정보나 결과를 짧고 구체적으로 쓴다.
- “이미지”, “스크린샷”처럼 보이는 사실만 반복하지 않는다.
- 주변 문장이 이미 같은 내용을 충분히 설명하면 중복 설명을 줄인다.
- 장식만을 위한 이미지는 가능하면 콘텐츠에 추가하지 않는다. 꼭 필요하고 아무 정보도 전달하지 않는다면 빈 alt를 사용한다.

### 개인정보와 정리

- 업로드 전에 이름, 이메일, 프로필 사진, 저장소명, 계정 ID, 알림, 브라우저 탭과 파일 경로를 확인한다.
- API key, token, 쿠키, 내부 URL, 고객·회사 정보와 비공개 코드가 보이면 가림 처리 후 다시 확인한다.
- 모바일 사진은 EXIF와 위치 정보 등 불필요한 메타데이터를 제거하거나 안전한 변환본을 사용한다.
- 글을 삭제해도 이미지는 자동 삭제되지 않는다. 저장소 전체에서 이미지 경로를 검색해 참조가 없는지 확인한 뒤 삭제한다.
- 삭제한 이미지는 Git 기록으로 복구할 수 있도록 이미지 삭제 commit을 다른 변경과 구분한다.

실제 입력과 배포 경로는 [Pages CMS 편집 안내](pages-cms-editor-guide.md#본문-이미지)를 함께 확인한다.
